## M2 — Size (Task 5): <bytes> vs 19,000,000 → <go / no-go>

**First measurement (V3 prune = old 5-define set):** litepdf.exe = **20,474,880 B**
vs 18,250,752 B at 1.24.11 → **+2,224,128 B (+2.12 MB)**; vs the 19,000,000 ceiling
→ **BREACH by 1,474,880 B**. Confirms the new default-ON features are the cost.
Applying spec D3 (prune-to-hold) below — adding levers one at a time:

| step | prune marker | added lever | litepdf.exe bytes | Δ vs prev | vs 19 MB |
|------|--------------|-------------|-------------------|-----------|----------|
| V3   | LITEPDF_PRUNE_V3 | (baseline: JS/OCR/DOCX off, TOFU_SYMBOL/CJK_LANG) | 20,474,880 | — | +1,474,880 OVER |
| V4   | LITEPDF_PRUNE_V4 | + FZ_ENABLE_BARCODE 0 | 20,474,880 | **0** | +1,474,880 OVER |
| V5   | LITEPDF_PRUNE_V5 | + FZ_ENABLE_BROTLI 0 + FZ_ENABLE_HYPHEN 0 + HYPHEN_ALL 0 | 19,353,088 | **-1,121,792** | +353,088 OVER |
| V6   | LITEPDF_PRUNE_V6 | + FZ_ENABLE_OFFICE 0 | 19,330,560 | -22,528 | +330,560 OVER |
| V7   | LITEPDF_PRUNE_V7 | + FZ_ENABLE_MOBI/HTML/TXT/IMG 0 (non-allowlist formats) | 19,309,056 | -21,504 | +309,056 OVER |
| V8   | LITEPDF_PRUNE_V8 | + TOFU_EMOJI | 19,309,056 | **0** | +309,056 OVER |
| V9   | LITEPDF_PRUNE_V9 | + TOFU_NOTO | **12,246,016** | **-7,063,040** | **−6,753,984 UNDER** |

**M2 conclusion — size IS holdable, but only via a blunt lever (product decision).**
- The feature toggles (barcode/brotli/hyphen/office/formats/emoji) bottom out at
  **19,309,056 B — still 309 KB over** — because ~1 MB of the 1.27 growth is core
  fitz/pdf with no toggle, and the only exe chunks left are the bundled fonts.
- `TOFU_NOTO` (drop ALL non-CJK Noto fallback fonts) is a **6.7 MB sledgehammer**
  — far more than the 309 KB needed, with no finer-grained Noto lever in MuPDF's
  TOFU system (all-or-nothing). It lands at **12.25 MB, 6.4 MB headroom**, keeping
  CJK (Source Han base + DroidSans) and Latin (Base14). Those Noto fallbacks
  (~6.7 MB) were in 1.24.11 too; they only matter for **non-embedded** PDFs in
  non-Latin/non-CJK scripts (Arabic, Hebrew, Thai, Indic, …). Most PDFs embed
  fonts, so real-world impact is narrow — but it IS a multilingual-fallback
  regression, a **product call for the user**, NOT a free size win.
- Headroom is so large that with `TOFU_NOTO`, Stage 2 could even RE-ENABLE the
  functional sacrifices (brotli/hyphen/formats) and still sit ~13 MB — i.e. the
  Noto drop alone absorbs all of 1.27's growth and then some.
- **Net M2:** GO-conditional — size holds iff the user accepts dropping non-CJK
  Noto fallback. Otherwise (full fallback kept) 1.27.2 breaches 19 MB by 309 KB
  ⇒ D3 "can't ⇒ defer". Decision deferred to the post-M3/M4 escalation.

**After all safe new-feature levers, still +330,560 B (~323 KB) over.** Note the
exe is still 19,330,560 − 18,250,752 = **+1,079,808 B (~1.03 MB) above the
1.24.11 baseline** — i.e. ~1 MB of the growth is core fitz/pdf version growth
with no feature toggle. The 19 MB *absolute* gate needs only ~323 KB more, but
no remaining new-feature lever supplies it. Next lever class: non-PDF **document
format handlers** (see investigation below) — a behavior decision, not a free
toggle.

**Format-support investigation (which handlers are safe to prune):** litepdf is
NOT PDF-only. `src/core/Document.cpp:133` allowlists `.pdf .epub .cbz .xps .fb2
.svg`; `src/ui/MainWindow.cpp:787-792` drag-drop accepts pdf/epub/cbz/xps
("Phase 2 supported formats"). ⇒ **EPUB, CBZ, XPS, FB2, SVG are shipped product
features — their `FZ_ENABLE_*` MUST stay on** (and EPUB/FB2 keep `HTML_ENGINE`
on). CJK fonts must stay (CJK is supported; the M4 cjk fixtures would tofu).
Only formats NOT in the allowlist are safe to disable: `FZ_ENABLE_MOBI`,
`FZ_ENABLE_HTML` (standalone .html docs — EPUB doesn't need it, only HTML_ENGINE),
`FZ_ENABLE_TXT`, `FZ_ENABLE_IMG` (standalone image-as-document; CBZ uses the image
*codecs*, not this handler — verify in Stage 2). These are mostly HTML-engine-
shared, so expected to behave like OFFICE (small).

**Key insight (V4): `FZ_ENABLE_BARCODE 0` changed the exe by ZERO bytes** (config.h
re-pruned to V4 confirmed, exe relinked 13:30). The linker's `/OPT:REF` already
dead-strips barcode — litepdf never calls `fz_*barcode*`, so it was never in the
exe regardless of the compile flag. ⇒ The +2.12 MB is NOT barcode; it is
*referenced* code/data (a decompression filter, hyphenation, fonts, or core
growth). Only levers whose code litepdf's call graph actually reaches shrink the
exe. (Barcode still bloats `libthirdparty.lib` build time / zint+zxing compile,
just not the final binary.)

## M3 — API + collector (Task 6): **GO (by API+source inspection; empirical test below)**

### Confirmed signatures (1.27.2 `structured-text.h`)
```c
// L630 — legacy collector, STILL PRESENT (what litepdf uses today, via fz_search_page)
int fz_search_stext_page(ctx, fz_stext_page *text, const char *needle,
                         int *hit_mark, fz_quad *hit_bbox, int hit_max);

// L1074 — NEW case-aware matcher: == fz_search_stext_page + a trailing `options`.
//          SAME marks/quads/max → count collector convention.
int fz_match_stext_page(ctx, fz_stext_page *text, const char *needle,
                        int *hit_mark, fz_quad *hit_bbox, int hit_max,
                        fz_search_options options);

// L1089 — NEW callback matcher for true mid-page cancel
int fz_match_stext_page_cb(ctx, fz_stext_page *page, const char *needle,
                           fz_match_callback_fn *cb, void *opaque,
                           fz_search_options options);

// L650 — callback: return 0 to continue, 1 to ABORT (L637/647)
typedef int (fz_match_callback_fn)(ctx, void *opaque, int num_quads,
                                   fz_quad *hit_bbox, int chapter, int page);

// L916 — options enum
typedef enum {
  FZ_SEARCH_EXACT = 0,            // case-SENSITIVE  ← Tier-1
  FZ_SEARCH_IGNORE_CASE = 1,      // case-insensitive (today's behavior)
  FZ_SEARCH_IGNORE_DIACRITICS = 2,
  FZ_SEARCH_REGEXP = 4,           // Tier-2 (deferred)
  FZ_SEARCH_KEEP_LINES = 8, FZ_SEARCH_KEEP_PARAGRAPHS = 16, FZ_SEARCH_KEEP_HYPHENS = 32
} fz_search_options;
```
Also: `util.h` page-level wrappers `fz_match_page(_number)(_cb)(…, options)` (L147-162),
and a streaming `fz_new_search`/`fz_feed_search`/`fz_search_set_options` incremental API.

### Collector feasibility
- **Best-case confirmed:** `fz_match_stext_page` keeps the exact `int* hit_mark,
  fz_quad* hit_bbox, int hit_max → int count` shape litepdf's `page_hits` already
  fills — Stage 3 migration is "add an `options` arg", zero structural change.
- **Cap-at-max return contract (Task 6 Step 4):** `fz_match_stext_page`
  (stext-search.c:1245) delegates to the `_cb` form via `oldsearch_cb`
  (stext-search.c:1206-1226), which fills the caller's array and **breaks at
  `fill == max_quads`** (L1215), returning `data.fill = min(actual, max_quads)`.
  The collector callback deliberately never returns 1 even when full (L1223-1224)
  so the count saturates exactly at `max_quads`. ⇒ **`n == kMaxQuads` ⟺
  truncation**, identical to 1.24.11's `fz_search_page` — production
  `Document.cpp:514` truncation logic needs NO change.
- **Case-sensitivity (Tier-1):** `FZ_SEARCH_EXACT` vs `FZ_SEARCH_IGNORE_CASE`,
  passed as the `options` arg. Direct enable for the spec's case-sensitive search.
- **Mid-page cancel (Tier-1):** see the empirical caveat below — the `_cb`
  wrapper does NOT honor the abort return in 1.27.2; the incremental `fz_search`
  loop does.

### Empirical proof (throwaway `tests/unit/spike_mupdf127_collector.cpp`)
Test wiring validated: the spike `#include`s `mupdf/fitz.h` and links — proving
`target_link_libraries(litepdf_unit_tests PRIVATE litepdf::mupdf)` propagates the
header dir (the plan-gate's PRIVATE-link concern). Reconfigure picked up the new
source; **ctest ran 4 tests (not 0)**. `ctest -R spike` → **4/4 PASS**:
1. `fz_match_stext_page(IGNORE_CASE)` "lorem" → **12** hits collected into
   `quads[]`/`marks[]` — the PageHit collector convention holds.
2. `fz_match_stext_page(EXACT)` "lorem" → **0** (fixture has "Lorem") — case
   sensitivity works as the Tier-1 toggle.
3. `fz_match_stext_page_cb` requesting abort-after-3 → **returns 12, NOT 3**.
   **The callback abort does NOT work in 1.27.2**: `fz_match_stext_page_cb`
   (stext-search.c:1158) discards the callback's return value and always runs to
   `FZ_SEARCH_COMPLETE`, despite the header doc (L637/647) saying 0=continue/
   1=abort. (Bonus upstream bug: L1158 indexes `quads[i]` after the fill loop —
   `i == num_quads`, an OOB read for the chapter/page args.)
4. Incremental loop (`fz_new_search` → `fz_search_set_options` →
   `fz_feed_search` → loop `fz_search_forwards`, break on the 3rd
   `FZ_SEARCH_MATCH`) → **seen == 3**. **True mid-page cancel IS achievable** via
   the lower-level API where the caller owns the loop.

**Stage 3 design consequence:** case-sensitive search = a drop-in `options` arg on
`fz_match_stext_page` (trivial). Mid-page `abort_flag` honoring must NOT use
`fz_match_stext_page_cb` (its abort is a no-op in 1.27.2); use the incremental
`fz_search` loop and check `abort_flag` between `fz_search_forwards` calls. More
code than the spec assumed, but proven to work. **M3 = GO.**

## Overall recommendation: **GO-AFTER-PRUNE — gated on one product decision (font fallback)**

All four measurements pass on their technical merits; the upgrade is viable. One
decision is the user's to make.

- **M1 build = GO.** 1.27.2 builds clean (/MT patch covers the new barcode/zxing
  vcxproj; 0 link errors).
- **M3 search = GO (the upgrade's whole point lands).** Case-sensitive search is
  a drop-in `options` arg on `fz_match_stext_page` (same collector convention,
  same cap semantics as 1.24.11 — `Document.cpp` truncation logic unchanged).
  Mid-page cancel works via the incremental `fz_search` loop (the
  `fz_match_stext_page_cb` callback-abort is a 1.27.2 no-op — design around it).
  All 4 spike tests green.
- **M4 determinism = GO with a known shift.** CJK byte-identical; large.pdf a
  benign Base14 sub-pixel delta. Stage 2 regenerates the Latin reference.
- **M2 size = the gating decision.** 1.27.2 carries ~1 MB of core growth with no
  feature toggle. Pruning every safe feature (barcode/brotli/hyphen/office/
  non-allowlist formats) bottoms out at **19,309,056 B — 309 KB OVER** the 19 MB
  gate. The only lever big enough is `TOFU_NOTO` (drop all non-CJK Noto fallback
  fonts, ~6.7 MB) → **12,246,016 B, 6.4 MB headroom**. CJK + Latin keep rendering
  correctly (proven in M4); the cost is **non-embedded** PDFs in non-Latin/non-CJK
  scripts (Arabic, Hebrew, Thai, Indic, …) would render tofu. Those fallback
  fonts shipped in 1.24.11 too, so this is a *reduction of existing* multilingual
  fallback, not a 1.27 regression. Spec D3 forbids raising the gate.

**Decision required of the user (post-spike gate):**
- **A — GO-AFTER-PRUNE (accept `TOFU_NOTO`):** ship 1.27.2 at ~12.25 MB, dropping
  non-CJK Noto fallback. Big headroom, Tier-1 search lands. Cost: non-embedded
  exotic-script PDFs lose fallback glyphs.
- **B — GO-AFTER-PRUNE (finer font pruning, Stage-2 investigation):** keep common
  fallback scripts, custom-patch MuPDF's noto.c font tables to drop only rare
  scripts to claw back ~400 KB. Preserves more coverage; more Stage-2 work, and
  unproven it can hit the target cleanly.
- **C — NO-GO-DEFER (strict D3):** keep full multilingual fallback, defer the
  1.27 upgrade until size holds without a font tradeoff. Tier-1 search waits.
