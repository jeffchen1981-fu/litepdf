## 1. Goal

Two orthogonal, positively-coupled improvements toward v1.0 polish:

1. **Shrink the binary** from 39.6 MB toward the design's 8 MB target by pruning
   MuPDF features litepdf never uses, the dominant lever being the 23.66 MB
   embedded Source Han Serif CJK font.
2. **Understand cold-start** by adding sub-span instrumentation that splits the
   currently-opaque `T1→T2` window (245–608 ms, the largest cold-start cost),
   so any future tuning targets a measured bottleneck instead of a guess.

The two couple: if the first `fz_new_context` indexing the 23.66 MB CJK font is
a material part of `T1→T2`, pruning the font improves both size **and**
cold-start. The instrumentation (built first) measures this before/after.

**Non-goal:** This phase does not commit to *hitting* 8 MB or to *any* specific
cold-start speedup. It commits to (a) removing unused features, (b) a
data-driven CJK font decision validated against real CJK PDFs, and (c) making
`T1→T2` measurable. Optimization beyond pruning is gated on what the
instrumentation reveals.

## 2. Measured Baseline (collected 2026-06-07, dev machine, real GPU)

The numbers that drive every decision below. Collected from the existing
`build/Release` artifacts via `scripts/benchmark.ps1` (headless) and
`litepdf.exe --log-timings` (GUI), `git_sha=c819464`.

### 2.1 Headless CPU path (best-of-9 min, `scripts/benchmark.ps1`)

| fixture | open_ms | engine_init_ms | render_ms | **open_render_ms** | first-sample (cold) |
|---------|---------|----------------|-----------|--------------------|---------------------|
| large.pdf (500 pp) | 0.74 | 0.24 | 6.55 | **7.54** | 56.2 |
| simple.pdf (15 KB) | 0.84 | 0.34 | 7.30 | **8.57** | 20.0 |

**The entire CPU document path is ~7.5 ms** — under 3% of cold-start. The cold
first-sample (20–56 ms) is the one-time `fz_new_context` cost. Blindly tuning
open/render is pointless; this is why D5's "gate-only" was right about the CPU
path, and why the real cold-start cost must be elsewhere.

### 2.2 GUI cold-start per-span (`litepdf.exe simple.pdf --log-timings`, 3 runs)

Spans are adjacent deltas of the `T0→Tn` line emitted by
[`ColdStartTimer`](../../../src/ui/ColdStartTimer.cpp).

| Span | meaning | run1 (cold) | run2 | run3 (warm) |
|------|---------|-------------|------|-------------|
| T0→T1 | wWinMain → ShowWindow returned | 381 | 39 | 32 |
| **T1→T2** | **→ doc open done (`WM_USER_OPEN_OK`)** | **441** | **471** | **245** |
| T2→T3 | → first page rendered | 56 | 26 | 25 |
| T3→T4 | → first real bitmap on screen | 5 | 6 | 1 |
| **T0→T4 total** | | **883** | **542** | **303** |

Cross-check with **large.pdf** GUI cold-start (T1→T2 = 316–608 ms): a 500-page
document has essentially the **same** `T1→T2` as a 1-page one. Combined with the
0.74 ms headless open, this proves **`T1→T2` is dominated by fixed first-time
initialization, not PDF work**.

**Two findings that overturn Phase 11's D5 assumption:**
- First-launch cold-start is **~883 ms**, not the cited 233 ms (that was a
  cherry-picked warm number). Real headroom to the 1 s target is ~1.1×, not 4×.
- The bottleneck is the fixed `T1→T2` init (D2D/GPU driver / MuPDF global init /
  font indexing), invisible to the current 5-point timer.

### 2.3 Binary composition (`.obj` sizes — strong proxy for fonts, which are
strongly-referenced data blobs; weak proxy for code, which the linker dead-strips)

| component | .obj size | prune effect |
|-----------|-----------|--------------|
| **Source Han Serif (CJK)** | **23.66 MB** | **the size lever** — strongly referenced → ~all in exe |
| DroidSansFallbackFull + Full | 4.84 + 3.39 MB | CJK fallback; dropped only in option C |
| Noto historic/symbol (Tangut, Cuneiform, Hieroglyphs…) | ~4–5 MB total | low-risk free size (`TOFU_HISTORIC`/`TOFU_SYMBOL`) |
| Tesseract OCR | 202 MB obj | mostly dead-stripped already (no OCR API calls); pruning saves compile time, little exe |
| mujs (pdf-js) | 203 KB | small; `FZ_ENABLE_JS=0` also removes sibling JS objs |

**Truth:** the exe is ~30 MB+ embedded fonts; MuPDF code is only ~5–8 MB.
Reaching ~8–10 MB **requires** dropping the CJK font. Tesseract/mujs are
rounding error in the exe.

## 3. Resolved Decisions (brainstorming 2026-06-07)

| # | Decision | Choice | Rationale |
|---|----------|--------|-----------|
| D1 | Measure or tune first? | **Measure first, then decide** | The CPU path is 7.5 ms; without per-span data, "cold-start tuning" is guesswork. (Q1) |
| D2 | Which axes in 11.5? | **Both: size pruning + cold-start instrumentation** | Orthogonal, non-blocking, positively coupled via the CJK font. Idle-RAM and disk-IO gating stay deferred (Phase 11 §7). (Q2) |
| D3 | CJK font strategy | **C with verification gate: drop embedded CJK (`TOFU_CJK`), rely on Windows system-font fallback; fall back to B (keep DroidSansFallback) if real CJK PDFs render tofu** | C reaches ~10 MB (near the 8 MB target) and most helps cold-start if the font is the init cost; B is the safety net. User is a zh-TW user — CJK PDFs are a real use case, so fallback MUST be validated, not assumed. (Q3) |
| D4 | Execution order | **Instrument-first (Approach 1)** | Honors D1: build the sub-span timer, capture a before baseline proving the CJK font is the `T1→T2` cost, then prune, then re-measure after with the same probes. C→B gate stands on data. |
| D5 | Historic/symbol Noto fonts | **Drop unconditionally** (`TOFU_HISTORIC`, `TOFU_SYMBOL`) | ~4–5 MB free size, near-zero risk (no one opens a Tangut/cuneiform PDF in a lightweight viewer). Independent of the CJK decision. |

## 4. Components

### 4.1 Axis 2 (built first) — cold-start sub-span instrumentation

Extend [`ColdStartTimer`](../../../src/ui/ColdStartTimer.hpp) to split the opaque
`T1→T2` window. The current timer is a fixed 5-slot global (`T0..T4`); add
named intermediate marks **inside** `T1→T2` without disturbing the existing
T0–T4 contract (smoke-test parses the same `T0→T4` line).

**The probe set must distinguish two candidate delays** (the core design point):

- **(a) open-thread work** — measured *on the worker thread* in
  [`open_tab_async`](../../../src/ui/MainWindow.cpp:140), relative to T0:
  - `tOpenStart` — thread entered, before `doc.open()`
  - `tOpenDone` — after `doc.open()` (first `fz_new_context` cost lands here)
  - `tViewBuilt` — after the `DocumentView` ctor (RenderEngine + workers +
    SearchSession), immediately before `PostMessageW(WM_USER_OPEN_OK)`
- **(b) UI-thread pump delay** — `T2` is when the UI thread *dequeues*
  `WM_USER_OPEN_OK`. If `tViewBuilt` (when the message was *posted*) is far
  earlier than `T2`, the UI thread was busy (first D2D factory init in a blank
  `WM_PAINT`) and the delay is **not** open work. If `tViewBuilt ≈ T2`, the open
  work itself is slow.

This `(tViewBuilt vs T2)` gap is the load-bearing measurement: it tells us
whether to optimize MuPDF init (open thread) or the first-paint/D2D path (UI
thread). Emit all sub-marks on the existing `--log-timings` line (extend the
format; keep `T0→T4` fields present for smoke-test back-compat).

**Constraint:** the worker-thread marks record QPC snapshots into the timer's
globals; `ColdStartTimer` is already a process-global singleton with idempotent
marks, so cross-thread writes to distinct slots are safe (each slot written once,
by one thread, before the UI thread reads on T4 emit). Document the threading
assumption in the header.

Deliverable: a before baseline (this build) and, after Axis 1, an after baseline
— same probes, same fixtures.

### 4.2 Axis 1 — MuPDF feature + font pruning

Inject a pruned MuPDF config at configure time, reusing the existing
[`ImportMuPDF.cmake`](../../../cmake/ImportMuPDF.cmake:65) vcxproj-patch pass
(the one that already rewrites `/MD→/MT`). Add `PreprocessorDefinitions` to the
MuPDF-compiling project(s) so the upstream `config.h` `#ifndef` guards keep the
injected values:

- **Feature flags (viewer never uses):** `FZ_ENABLE_JS=0`, `FZ_ENABLE_XPS=0`,
  `FZ_ENABLE_SVG=0`, `FZ_ENABLE_OCR_OUTPUT=0`, `FZ_ENABLE_DOCX_OUTPUT=0`.
  Keep `FZ_ENABLE_HTML`/`FZ_ENABLE_EPUB` (ePub) and `FZ_ENABLE_CBZ` (CBZ).
  Closing `OCR_OUTPUT` also disables the Tesseract/Leptonica path
  (`config.h:216`).
- **Fonts:** `TOFU_CJK` (drop Source Han Serif + DroidSansFallback — D3 option C),
  `TOFU_HISTORIC`, `TOFU_SYMBOL` (D5). Do **not** set `TOFU_SIL` (ePub/HTML body
  text needs Charis SIL) or `TOFU_BASE14` (core PDF text).

**Mechanism note for the plan to verify:** `TOFU_CJK` affects `font.c` (which
decides *which* font symbols to reference); the embedded font bytes live in the
`libresources` project. Dropping the reference lets the linker dead-strip the
CJK `.obj`s. The plan must confirm the define reaches the `font.c` compilation
unit and that the resulting exe actually shrinks (measure, don't assume).

**Open mechanism choice (plan-level):** inject via vcxproj
`PreprocessorDefinitions` patch (consistent with the existing `/MT` patch) vs.
patching `config.h` directly. Both modify submodule files at configure time;
prefer the vcxproj patch for locality with the existing pass unless it proves
not to reach `font.c`.

### 4.3 Axis 1 verification gate — C→B fallback

The risk in D3 option C: dropping embedded CJK assumes Windows system-font
fallback ([`font.c`](../../../third_party/mupdf/source/fitz/font.c) Windows path)
covers CJK PDFs. Validate, don't assume:

- **Fixtures:** add small CJK PDF fixtures — at minimum zh (Traditional),
  ja, ko — with embedded-font-free text (so the viewer must fall back to a
  system font). Generated deterministically like the other fixtures
  (`scripts/generate-*-fixture.py`), zlib-independent (`pageCompression=0`,
  per project memory `reference_litepdf_reportlab_zlib_determinism`).
- **Check:** open each under the pruned build; assert glyphs render (no tofu
  boxes). Headless-checkable via the CLI render path
  (`litepdf-cli <fixture> --render 0` → non-blank pixmap with CJK coverage),
  or a smoke-test launch asserting first-paint succeeds.
- **Fallback rule:** if any CJK fixture renders tofu under C, **drop `TOFU_CJK`**
  and ship B (keep DroidSansFallback, ~3.4 MB, still offline-CJK-capable). The
  feature-flag prunes and historic/symbol drops stay regardless.

### 4.4 Size-gate update (coupled to Phase 11)

The Phase 11 gate's `exe_bytes` check is anti-regression (256 KB tolerance)
against the PR's rebuilt base. After pruning, the base/PR builds both shrink, so
the relative gate keeps working with no change. Also add
an **absolute ceiling** to `benchmarks/thresholds.json` (e.g.
`size_ceiling_bytes` ≈ 12 MB, sized off the post-prune measured exe + headroom)
so a future change that re-bloats the binary (e.g. re-adding a large font) fails
CI even if it slips under the +256 KB per-PR delta. This operationalizes the
8 MB-target intent that Phase 11 explicitly deferred to 11.5. **Resolved (§9.1):
add it.**

## 5. Error Handling / Fallback

- **CJK fallback fails (tofu):** deterministic C→B downgrade per §4.3. Not a
  build error — a documented decision branch with a fixture-driven trigger.
- **Pruned build link errors** (a `FZ_ENABLE_*=0` that some kept path still
  references): treat as a signal the feature isn't actually unused; re-enable
  that one flag, record why in the spec. Fail closed (don't ship a half-pruned
  build that crashes on a format).
- **Sub-span probe on a failed open:** if `doc.open()` fails, the open-thread
  marks past `tOpenStart` are never set; the timer already guards unset slots
  (`elapsed_ms` returns 0). The `--log-timings` line must not print garbage for
  an aborted open — emit only on the complete T0–T4 path as today.

## 6. Testing Strategy

- **Instrumentation:** a unit-level check that `ColdStartTimer` sub-marks are
  monotonic and that unset slots report 0 (mirrors the existing idempotent-mark
  contract). The real output is the before/after `--log-timings` comparison,
  recorded in the PR description as evidence.
- **Pruning:** the benchmark gate's exe_bytes delta proves the shrink; the
  harness smoke (`litepdf-cli large.pdf --benchmark --json`) proves the pruned
  MuPDF still opens+renders PDFs.
- **CJK gate:** the §4.3 fixtures, run under the pruned build, are the
  exit-criterion evidence for the C-vs-B decision.
- **Regression safety net:** `smoke-test.ps1`'s `T0→T4 ≤ 1500 ms` absolute
  ceiling stays as-is; it already covers a gross cold-start blowout. **Resolved
  (§9.2):** do NOT add a new cold-start regression gate — warm/cold variance
  (303→883 ms) is too wide to gate at ±%, and the headless CPU gate already
  protects the part that's stable. Keep cold-start as a measured, smoke-ceilinged
  number, not a ±% gate.

## 7. What This Phase Does NOT Do

- Hit 8 MB as a hard requirement (target, not contract — measure how close
  pruning gets).
- Optimize cold-start beyond what pruning incidentally buys, unless the
  instrumentation reveals a cheap, high-leverage fix (then it's a follow-up
  decision, not assumed scope).
- Idle-RAM or disk-IO (cold-cache) gating (still deferred, Phase 11 §7).
- The MuPDF 1.27+ upgrade (post-v1.0, unrelated to pruning).
- Re-architect `ColdStartTimer` into a general tracing system — minimal
  sub-marks for `T1→T2` only.

## 8. Build Sequence (for the implementation plan)

1. **Axis 2 first:** extend `ColdStartTimer` with `T1→T2` sub-marks (open-thread
   `tOpenStart`/`tOpenDone`/`tViewBuilt` + UI-thread `T2` gap); wire into
   `open_tab_async` + the `--log-timings` emit. Build, capture the **before**
   baseline (simple.pdf + large.pdf, several runs); record the
   `tViewBuilt-vs-T2` gap to locate the bottleneck.
2. **Axis 1 config injection:** patch `ImportMuPDF.cmake` to inject the §4.2
   feature flags + `TOFU_HISTORIC`/`TOFU_SYMBOL` first (low-risk), rebuild,
   measure exe delta + harness smoke.
3. **Axis 1 CJK (option C):** add `TOFU_CJK`, rebuild, measure exe + the
   **after** cold-start baseline with the §4.1 probes.
4. **CJK gate:** add zh/ja/ko fixtures + the §4.3 render check; run under the
   pruned build. If tofu → drop `TOFU_CJK` (ship B); else keep C.
5. **Size gate:** update `benchmarks/thresholds.json` baseline; add the absolute
   ceiling if §4.4 is approved.
6. **Docs:** update the roadmap Phase 11.5 row + known-limitations (final exe
   size, CJK strategy shipped, before/after cold-start), and reconcile the
   Phase 11 spec's "deferred to 11.5" pointers now that 11.5 is done.

## 9. Resolved Decisions (user review, 2026-06-07)

All three accepted as proposed:

1. **Size absolute ceiling (§4.4):** **add** `size_ceiling_bytes` (~12 MB) to the
   gate, sized off the post-prune exe + headroom. Catches a future re-bloat that
   slips under the per-PR +256 KB delta.
2. **Cold-start gate (§6):** **no** new ±% cold-start regression gate; the
   1500 ms `smoke-test.ps1` ceiling stays the only cold-start guard (warm/cold
   variance 303→883 ms is too wide to gate at ±%).
3. **CJK fixture scope (§4.3):** **zh + ja + ko** — system-font fallback breaks
   per-script, not globally, so all three are required coverage.
