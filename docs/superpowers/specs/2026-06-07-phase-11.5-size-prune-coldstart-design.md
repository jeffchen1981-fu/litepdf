## 1. Goal

Two orthogonal, positively-coupled improvements toward v1.0 polish:

1. **Shrink the binary** from 39.6 MB by pruning MuPDF features litepdf never
   uses and downgrading the embedded CJK font from the 23.66 MB Source Han Serif
   to the ~4.8 MB Droid Sans Fallback Full (still fully offline-CJK-capable).
   Target: **~16–18 MB** (measured, not assumed). The 8 MB target needs a
   Windows system-font loader (a real feature) and is **deferred to post-v1.0**
   (§7).
2. **Understand cold-start** by adding sub-span instrumentation that splits the
   currently-opaque `T1→T2` window (245–608 ms, the largest cold-start cost),
   so any future tuning targets a measured bottleneck instead of a guess.

The two couple: if the first `fz_new_context` indexing the 23.66 MB CJK font is
a material part of `T1→T2`, downgrading the font improves both size **and**
cold-start. The instrumentation (built first) measures this before/after.

**Non-goal:** This phase does not commit to *hitting* 8 MB or to *any* specific
cold-start speedup. It commits to (a) removing unused features, (b) a verified
CJK-font downgrade that still renders CJK, and (c) making `T1→T2` measurable.

## 2. Measured Baseline (collected 2026-06-07, dev machine, real GPU)

The numbers that drive every decision below. Collected from the existing
`build/Release` artifacts via `scripts/benchmark.ps1` (headless) and
`litepdf.exe --log-timings` (GUI), `git_sha=c819464`.

### 2.1 Headless CPU path (best-of-9 min, `scripts/benchmark.ps1`)

| fixture | open_ms | engine_init_ms | render_ms | **open_render_ms** | first-sample (cold) |
|---------|---------|----------------|-----------|--------------------|---------------------|
| large.pdf (500 pp) | 0.74 | 0.24 | 6.55 | **7.54** | 56.2 |
| simple.pdf (15 KB) | 0.84 | 0.34 | 7.30 | **8.57** | 20.0 |

**The entire CPU document path is ~7.5 ms** — under 3% of cold-start. The cold
first-sample (20–56 ms) is the one-time `fz_new_context` cost. Blindly tuning
open/render is pointless; this is why D5's "gate-only" was right about the CPU
path, and why the real cold-start cost must be elsewhere.

### 2.2 GUI cold-start per-span (`litepdf.exe simple.pdf --log-timings`, 3 runs)

Spans are adjacent deltas of the `T0→Tn` line emitted by
[`ColdStartTimer`](../../../src/ui/ColdStartTimer.cpp).

| Span | meaning | run1 (cold) | run2 | run3 (warm) |
|------|---------|-------------|------|-------------|
| T0→T1 | wWinMain → ShowWindow returned | 381 | 39 | 32 |
| **T1→T2** | **→ doc open done (`WM_USER_OPEN_OK`)** | **441** | **471** | **245** |
| T2→T3 | → first page rendered | 56 | 26 | 25 |
| T3→T4 | → first real bitmap on screen | 5 | 6 | 1 |
| **T0→T4 total** | | **883** | **542** | **303** |

Cross-check with **large.pdf** GUI cold-start (T1→T2 = 316–608 ms): a 500-page
document has essentially the **same** `T1→T2` as a 1-page one. Combined with the
0.74 ms headless open, this proves **`T1→T2` is dominated by fixed first-time
initialization, not PDF work**.

**Two findings that overturn Phase 11's D5 assumption:**
- First-launch cold-start is **~883 ms**, not the cited 233 ms (that was a
  cherry-picked warm number). Real headroom to the 1 s target is ~1.1×, not 4×.
- The bottleneck is the fixed `T1→T2` init, invisible to the current 5-point timer.

### 2.3 Binary composition (`.obj` sizes — strong proxy for fonts, which are
strongly-referenced data blobs via the `inbuilt_fonts[]` table in `noto.c`; weak
proxy for code, which the linker dead-strips)

| component | .obj size | prune effect |
|-----------|-----------|--------------|
| **Source Han Serif (CJK, current)** | **23.66 MB** | **the size lever** — downgraded to Droid Sans Fallback Full (4.84 MB) via `TOFU_CJK_LANG` (D3) |
| Droid Sans Fallback Full | 4.84 MB | the B-trim replacement CJK font (referenced once `TOFU_CJK_LANG` is set) |
| Noto historic/symbol (Tangut, Cuneiform, Hieroglyphs…) | ~4–5 MB total | low-risk free size (`TOFU_HISTORIC`/`TOFU_SYMBOL`, D5) |
| Tesseract OCR | 202 MB obj | **exe impact unmeasured** — likely mostly dead-stripped (no OCR API calls), but `HAVE_TESSERACT` is set in the vcxproj so OCR code compiles; pruned for hygiene/compile-time, exe saving TBD by measurement |
| mujs (pdf-js) | 203 KB | small; `FZ_ENABLE_JS=0` also removes sibling JS objs |

**Truth:** the exe is ~30 MB+ embedded fonts; MuPDF code is only ~5–8 MB.
Reaching ~8 MB needs dropping CJK *entirely* (a Windows system-font loader,
post-v1.0). B-trim (Source Han Serif → Droid Sans Fallback Full) lands ~16–18 MB
while keeping offline CJK — the achievable v1.0 win.

## 4. Components

### 4.1 Axis 2 (built first) — cold-start sub-span instrumentation

Extend [`ColdStartTimer`](../../../src/ui/ColdStartTimer.hpp) to split the opaque
`T1→T2` window. The current timer is a fixed 5-slot global (`T0..T4`); add named
intermediate marks inside `T1→T2` without disturbing the existing T0–T4 contract
(smoke-test parses the same `T0→T4` line).

**Probe set** (relative to T0): on the worker thread in
[`open_tab_async`](../../../src/ui/MainWindow.cpp:140) — `tOpenStart` (thread
entered), `tOpenDone` (after `doc.open()` — first `fz_new_context` cost lands
here), `tViewBuilt` (after the `DocumentView` ctor, immediately before
`PostMessageW(WM_USER_OPEN_OK)`). The existing `T2` is when the UI thread
*dequeues* that message.

**The `(tViewBuilt vs T2)` gap is the load-bearing measurement:** it splits
"open work genuinely slow" (gap small → optimize MuPDF init on the worker) from
"UI thread not pumping the queue" (gap large → the UI thread is busy between
ShowWindow and dequeue). **Do NOT pre-attribute the UI-thread case to a specific
cause.** v1 wrongly blamed "first D2D factory init in a blank WM_PAINT", but
`D2D1CreateFactory` runs in the `PdfCanvas` ctor inside `WM_CREATE`
([`PdfCanvas.cpp:313`](../../../src/ui/PdfCanvas.cpp:313),
[`MainWindow.cpp:651`](../../../src/ui/MainWindow.cpp:651)) — **before** ShowWindow/T1
— so that cost is already in `T0→T1` (Opus MAJOR-4). Honest UI-thread candidates
to enumerate when the gap is large: first `ID2D1HwndRenderTarget`/GPU-device
creation on the first real `WM_PAINT`, the synchronous `UpdateWindow` paint, and
generic message-queue latency. The instrumentation *locates* the cost; it does
not assume it. If `T0→T1` itself is large, add a sub-mark there too (D2D factory
ctor timing).

**Thread-safety (correct rationale):** `ColdStartTimer`'s globals (`g_t[5]`,
`g_marked[5]`) carry no atomics/fences. Cross-thread visibility is provided **by
the `PostMessageW`→`GetMessage` happens-before edge** (kernel-mediated full
barrier): the worker writes all sub-marks *before* posting `WM_USER_OPEN_OK`, and
the UI thread reads them only after dequeuing it. The existing 5-point design
relies on the same property (T2/T3/T4 marked on the UI thread post-dispatch).
Document this in the header — single-writer-per-slot is necessary but **not
sufficient**; the message handoff is the load-bearing guarantee, so a future
read on a thread that didn't cross that edge would be unsynchronized. The
`mark()` check-then-set is non-atomic and fine only while each slot has one writer.

Deliverable: a before baseline (this build) and, after Axis 1, an after baseline
— same probes, same fixtures.

### 4.2 Axis 1 — MuPDF feature + font pruning

Inject the pruned config by **patching `third_party/mupdf/include/mupdf/fitz/config.h`
directly** at configure time (single edit, applies uniformly to every translation
unit, and the file's `#ifndef X / #define X 1` guards mean a prepended `#define`
survives). This is preferred over the vcxproj `PreprocessorDefinitions` route:
those lines are **not uniform** across the 6 Config|Platform blocks of
`libmupdf.vcxproj`, so the existing blind `string(REPLACE)` /MT pass does **not**
generalize to them (Opus MAJOR-3). The patch is still applied in
[`ImportMuPDF.cmake`](../../../cmake/ImportMuPDF.cmake) at configure time, next to
the existing /MT rewrite.

Defines to inject:

- **Feature flags (grep-clean in `src/`, D6):** `FZ_ENABLE_JS=0`,
  `FZ_ENABLE_OCR_OUTPUT=0`, `FZ_ENABLE_DOCX_OUTPUT=0`. **Keep**
  `FZ_ENABLE_XPS`, `FZ_ENABLE_SVG`, `FZ_ENABLE_HTML`, `FZ_ENABLE_EPUB`,
  `FZ_ENABLE_CBZ`.
- **Fonts:** `TOFU_CJK_LANG` (Source Han Serif → Droid Sans Fallback Full — D3;
  do **NOT** use `TOFU_CJK`, which removes all CJK and needs a system loader),
  `TOFU_HISTORIC`, `TOFU_SYMBOL` (D5). Do **not** set `TOFU_SIL` (ePub/HTML body
  text needs Charis SIL) or `TOFU_BASE14` (core PDF text).

**Verification target is `noto.c`, not `font.c`** (Opus MAJOR-3, Sonnet F4):
`TOFU_CJK_LANG`/`TOFU_HISTORIC`/`TOFU_SYMBOL` gate the `FONT(...)` entries in
[`font-table.h`](../../../third_party/mupdf/source/fitz/font-table.h), which is
`#include`d by `noto.c` (compiled in `libmupdf.vcxproj`). `font.c` does not
include the table. The plan must confirm the define reaches the `noto.c` TU and
that the resulting exe actually shrinks (the unreferenced Source Han Serif `.obj`
in `libresources` is then dead-stripped). **Note on OCR (Opus NIT-8, Sonnet F11):**
`FZ_ENABLE_OCR_OUTPUT=0` disables the OCR *output writer* only; it does **not**
clear `HAVE_TESSERACT`/`HAVE_LEPTONICA` (set in the vcxproj Release|x64 defines).
Fully dropping the Tesseract link, if desired, additionally requires removing the
`libtesseract` ProjectReference / those defines — the plan must verify, not assume.

**Rebuild correctness (Sonnet F2 — BLOCKER):** `ExternalProject_Add` uses
`BUILD_ALWAYS OFF`, so it re-invokes MSBuild only when its **stamp** is absent,
not when the patched `config.h` changes on disk. After patching, the plan MUST
delete the stamp
(`Remove-Item build/mupdf_ext-prefix/src/mupdf_ext-stamp/Release/mupdf_ext-build -Force`)
or flip `BUILD_ALWAYS ON` for this phase — otherwise the rebuild silently no-ops
and the size measurement reflects the un-pruned binary.

### 4.3 Axis 1 verification — CJK still renders

B-trim keeps a real CJK font in the binary, so the failure mode is not tofu but
**incomplete coverage** (Droid Sans Fallback Full has fewer glyphs than Source
Han Serif, especially CJK-Ext). Verify, don't assume:

- **Fixtures:** add small CJK PDF fixtures — zh-Hant (the user's locale), ja, ko
  — generated deterministically (reportlab, `pageCompression=0`, pinned metadata,
  per project memory `reference_litepdf_reportlab_zlib_determinism`). Each
  fixture's text must use **CID/Unicode encoding without embedding a CJK font**
  (so rendering exercises MuPDF's inbuilt Droid Sans Fallback, not an embedded
  font — Sonnet F7); mirror `generate-search-fixture.py`'s CID approach but force
  the no-embed path.
- **Check (honest about mechanism — Opus MAJOR-5, Sonnet F3):** tofu-vs-glyph is
  **not** mechanically detectable via `litepdf-cli --render` (a page of `.notdef`
  boxes is a successful non-blank PPM; the CLI returns only pixmap/no-pixmap/
  timeout). So the CJK check is a **manual visual exit-criterion**: render each
  fixture under the pruned build, eyeball against a **checked-in reference PNG**,
  attach before/after screenshots to the PR. Automated part is liveness only
  (fixture opens + first paint succeeds, no crash). State this explicitly; do not
  imply `--render` settles glyph quality.
- **Coverage fallback:** if Droid Sans Fallback Full misses glyphs that matter
  for zh-Hant (verified via the fixtures), the escalation is C-real
  (post-v1.0 system-font loader), not a different macro — record the gap and
  ship B-trim with the known limitation documented.

### 4.4 Size-gate update (coupled to Phase 11)

The Phase 11 gate's `exe_bytes` check is anti-regression (256 KB tolerance)
against the PR's rebuilt base; after pruning, base/PR both shrink, so the
relative gate keeps working unchanged. Also add an **absolute ceiling**
(`size_ceiling_bytes` ≈ post-prune exe + headroom, e.g. ~20 MB) so a future
re-bloat that slips under the +256 KB per-PR delta still fails CI. **Resolved
(§9.1): add it.**

**This is a CODE change, not just JSON (Sonnet F5 — MAJOR):**
[`check-benchmark-regression.ps1`](../../../scripts/check-benchmark-regression.ps1)
validates exactly three threshold fields and enforces none named
`size_ceiling_bytes`. Adding the field to `benchmarks/thresholds.json` is inert
until the script is extended to (a) read it, (b) `Assert-PositiveFinite` it,
(c) fail when `prBytes > size_ceiling_bytes`; plus an **8th `-SelfTest`
assertion** covering the ceiling path. Any new PowerShell stays 5.1-parse-safe
(no `?.`/`??`/ternary — project memory `reference_litepdf_powershell_51_only`).

Also update [`smoke-test.ps1`](../../../scripts/smoke-test.ps1)'s `$maxBytes`
(currently `50MB`, comment says "~39 MB") to a post-prune value (~20 MB) — it
runs on every push/PR via `ci.yml` and is the more frequent size guard (Sonnet F8).

## 5. Error Handling (fail-closed)

- **CJK coverage gap:** documented known-limitation + post-v1.0 C-real escalation
  (§4.3), not a build failure.
- **Pruned build link errors** (a `FZ_ENABLE_*=0` that some kept path still
  references): signal the feature isn't actually unused — re-enable it, record
  why. Don't ship a half-pruned build that crashes on a format (§5 "fail closed").
- **Rebuild no-op:** the stale-stamp deletion (§4.2) is mandatory; a silent
  un-pruned rebuild would make the size measurement lie.
- **Sub-span probe on a failed open:** if `doc.open()` fails, marks past
  `tOpenStart` are never set; the timer already guards unset slots (`elapsed_ms`
  returns 0) and emits only on the complete T0–T4 path.

## 6. Testing Strategy

- **Instrumentation:** a unit check that `ColdStartTimer` sub-marks are monotonic
  and unset slots report 0. The real output is the before/after `--log-timings`
  comparison in the PR description.
- **Pruning:** the benchmark gate's exe_bytes delta proves the shrink; the
  harness smoke (`litepdf-cli large.pdf --benchmark --json`) proves the pruned
  MuPDF still opens+renders PDFs; the existing ePub/CBZ/XPS smoke launches prove
  the kept formats still work (XPS especially, since it was nearly dropped).
- **CJK:** the §4.3 manual visual check vs checked-in reference PNGs.
- **No new cold-start gate (Resolved §9.2):** `smoke-test.ps1`'s
  `T0→T4 ≤ 1500 ms` ceiling stays the only cold-start guard — warm/cold variance
  (303→883 ms) is too wide to gate at ±%, and the headless CPU gate already
  protects the stable part.

## 7. What This Phase Does NOT Do

- Hit 8 MB (needs C-real, below).
- **C-real (deferred to post-v1.0 / v1.1):** implement a Windows CJK system-font
  loader — `fz_install_load_system_font_funcs()` with a DirectWrite/GDI callback
  (`CreateFontW`/`GetFontData` → `fz_new_font_from_memory`) installed in
  [`Document::Impl`](../../../src/core/Document.cpp) — then set full `TOFU_CJK`
  to drop all embedded CJK and reach ~10 MB. This is a real feature with its own
  risks (which system font per script: MingLiU/MSGothic/Malgun; metrics fidelity;
  per-script coverage) and gets its own design + spec when scheduled.
- Optimize cold-start beyond what pruning incidentally buys, unless the
  instrumentation reveals a cheap high-leverage fix (then a follow-up decision).
- Idle-RAM or disk-IO gating (still deferred, Phase 11 §7).
- The MuPDF 1.27+ upgrade (post-v1.0, unrelated).
- Re-architect `ColdStartTimer` into a general tracer — minimal `T1→T2` sub-marks
  only.

## 8. Build Sequence (for the implementation plan)

1. **Axis 2 first:** extend `ColdStartTimer` with `T1→T2` sub-marks
   (`tOpenStart`/`tOpenDone`/`tViewBuilt` + the `tViewBuilt`-vs-`T2` gap; add a
   `T0→T1` D2D-ctor sub-mark too) and the corrected threading doc in the header;
   wire into `open_tab_async` + `--log-timings`. Build, capture the **before**
   baseline (simple.pdf + large.pdf, several runs); record the gap to locate the
   bottleneck. (Rebuild here is the litepdf exe only — no MuPDF change yet.)
2. **Axis 1 low-risk prune:** patch `config.h` via `ImportMuPDF.cmake` to inject
   `FZ_ENABLE_JS=0`, `FZ_ENABLE_OCR_OUTPUT=0`, `FZ_ENABLE_DOCX_OUTPUT=0`,
   `TOFU_HISTORIC`, `TOFU_SYMBOL`. **Delete the ExternalProject stamp**, rebuild
   MuPDF, measure exe delta + run the full smoke (incl. XPS/ePub/CBZ liveness)
   to confirm no kept format broke.
3. **Axis 1 CJK downgrade:** add `TOFU_CJK_LANG`; delete stamp; rebuild; measure
   exe + the **after** cold-start baseline with the §4.1 probes (the coupling
   test: did dropping the 23.66 MB font move `T1→T2`?).
4. **CJK verification:** add zh-Hant/ja/ko fixtures (+ `--check` determinism mode,
   wired into `benchmark.yml` alongside the existing `large.pdf --check` — Sonnet
   F6) + checked-in reference PNGs; manual visual confirm coverage. If zh-Hant
   coverage is inadequate → document known-limitation, flag C-real for post-v1.0.
5. **Size gate:** extend `check-benchmark-regression.ps1` to enforce
   `size_ceiling_bytes` (+ 8th self-test assertion); set the ceiling from the
   post-prune exe; update `smoke-test.ps1` `$maxBytes` + comment.
6. **Docs:** update the roadmap Phase 11.5 row + known-limitations (final exe
   size, CJK = Droid Sans Fallback Full, C-real deferred, before/after
   cold-start), and reconcile the Phase 11 spec's "deferred to 11.5" pointers.
