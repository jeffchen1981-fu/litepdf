## 1. Goal

Upgrade MuPDF from 1.24.11 to the latest **1.27.x** stable, then activate the
two search capabilities whose scaffolding is already wired but dormant:

- **Case-sensitive search** — make `SearchFlags::match_case` real via
  `fz_search_options` (`FZ_SEARCH_EXACT` / `FZ_SEARCH_IGNORE_CASE`).
- **True mid-page search cancellation** — honor `abort_flag` via the
  `fz_match_stext_page_cb` callback (return 1 to stop), replacing the
  run-to-completion behavior of 1.24.11's `fz_search_page`.

**Explicitly out of scope this effort (Tier 2, its own later phase):** regex
search (`FZ_SEARCH_REGEXP`), whole-word search (regex `\bword\b`), the FindBar
UI toggles those need, and the centered ~30-char snippet (phase-6.2). These all
require real UI design and warrant a separate brainstorm. The MuPDF 8 MB exe
target also stays deferred to post-v1.0.

## 2. Scope decisions (from brainstorm, 2026-06-08)

| # | Decision | Choice |
|---|----------|--------|
| D1 | Timing / intent | **Spike-first**, then full upgrade — convert the deferral's unknown-risks (build, size, API) into data before committing. |
| D2 | Search-feature scope | **Tier 1 only** (case-sensitive + mid-page cancel). Tier 2 (regex/whole-word + UI) deferred to its own phase. |
| D3 | Size-gate breach policy | If 1.27 breaches the 19 MB benchmark ceiling, **prune harder to claw back under**; can't prune back ⇒ upgrade is a **no-go / defer**. Do NOT raise the gate (it is Phase 11.5 load-bearing). |

## 3. Architecture — three independently verifiable stages

The load-bearing design choice is separating **Stage 2 (upgrade, behavior
identical)** from **Stage 3 (feature activation)**: if a determinism / size /
render gate breaks, it is attributable to the library bump, not the API
migration.

| Stage | Branch | Changes | Merge | Gate |
|-------|--------|---------|-------|------|
| 1. Spike | `spike/mupdf-1.27` (throwaway) | submodule bump + prune adaptation + measurement | **not merged**; findings captured | go/no-go |
| 2. Upgrade landing | `mupdf-1.27-upgrade` → PR | submodule pin + prune V3 + size-claw-back; **`src/` unchanged** | squash to `main` | 173 tests + exe < 19 MB + determinism + 3-lens |
| 3. Tier-1 activation | `search-tier1-activation` → PR | `page_hits` → stext matcher; wire `match_case` + `abort_flag` | squash to `main` | flipped + new search tests + 3-lens |

Stage 2 and Stage 3 are **two separate PRs** (cleaner independent review + git
bisect), not one PR with two commits.

## 4. Stage 1 — De-risk spike

Touches only `third_party/mupdf` (checkout the chosen 1.27.x tag) and
`cmake/ImportMuPDF.cmake` (prune adaptation, `_PRUNE_VER` → V3). **No `src/`
changes.** First step: confirm the latest 1.27.x stable tag upstream (the
roadmap names 1.27.2 / 2026-02, but today is 2026-06 — pin the newest 1.27.x
point release that is not known-broken, never an experimental/pre-release tag).

Four measurements, each with an explicit go/no-go:

1. **Build.** Does 1.27.x build cleanly on the MSVC `mupdf.sln` with the adapted
   prune, and does `litepdf` link?
   - ✅ go: clean `Release|Win32` build. ❌ no-go: sln structure / dependency
     changes in 1.27 that cannot be fixed cheaply.
   - **vcxproj/props inventory check** (3-lens: Sonnet I3 + Codex #4): the prune
     rewrites `/MD → /MT` only across `platform/win32/*.vcxproj` via GLOB
     (`ImportMuPDF.cmake:65,69`). 1.27 may add/rename projects or move runtime
     settings into `.props`. Diff the `*.vcxproj`/`*.props` inventory 1.24.11 vs
     1.27.x; after build, **scan every built project for a remaining
     `MultiThreadedDLL`** (a leftover `/MD` causes the `__imp_*` LNK2001
     documented at `ImportMuPDF.cmake:57-60`); verify the MSBuild project list
     and update `BUILD_BYPRODUCTS`/linked libs if the inventory changed.
2. **Size.** Measure `litepdf.exe` bytes vs the 19,000,000 ceiling (current
   18,250,752; ~731 KB headroom). Record **both** the raw 1.27 delta **and**
   each prune lever's individual delta (Codex #6 — the headroom is tight enough
   that per-lever attribution matters when clawing back).
   - Over ⇒ per D3, attempt additional pruning (new `FZ_ENABLE_*` or font
     levers) to get back under. Can't get under ⇒ **no-go / defer**.
3. **API + collector prototype** (3-lens High — Opus F1 + Sonnet C1 + Codex #1).
   Link-checking symbol *existence* is not enough: Stage 3 assumes 1.27's matcher
   maps cleanly onto today's `PageHit` contract (one quad per hit, `marks[]`
   grouping, `kMaxQuads = 256` cap, needle-only snippet — `Document.cpp:484,498,529`,
   `Document.hpp:102`). If 1.27 instead returns grouped/multi-quad results, a
   stateful search object, or callback result structs, Stage 3 is no longer a
   local call-replacement. So the spike must **compile and run a tiny throwaway
   collector** that:
   - resolves `fz_match_stext_page`, `fz_match_stext_page_cb`, `fz_search_options`
     as real symbols and **records their full actual signatures** in the findings
     (not "resolved OK" — the exact callback shape: does it return `int` to abort,
     take `void* userdata` + a hit struct, etc.);
   - fills the existing `PageHit` shape from real hits and proves the
     `kMaxQuads`-cap + truncation contract is re-implementable on the new matcher;
   - proves callback-driven abort actually stops mid-page;
   - confirms `fz_search_page` still exists in 1.27 (Stage 2 keeps using it).
   - ❌ no-go: the matcher is absent at the pinned tag, its collection shape can't
     be mapped onto `PageHit` cheaply, or abort can't be wired.
4. **Render determinism.** Run the byte-identity `--check` on `large.pdf` +
   `cjk-{zh-hant,ja,ko}.pdf` under 1.27. Record **identical** or **shifted**
   only — do NOT fix it in the spike (keeps the spike a pure measurement). A
   MuPDF bump can plausibly change font hinting / anti-aliasing; this is the
   early warning that Stage 2 will need to regenerate reference fixtures.

**Output:** a short findings note (size delta, API status, determinism
identical/shifted, prune changes required) that gates Stage 2/3. The spike
branch is discarded after the findings are recorded.

## 5. Stage 2 — Upgrade landing (search behavior identical)

"Identical" here means **search source + output unchanged**, NOT necessarily
render-byte-identical: a MuPDF bump can legitimately shift raster output, so
reviewers must not reject a justified determinism-fixture regeneration as a
regression (Codex #3).

**Files:**

- `third_party/mupdf`: `git checkout <1.27.x tag>`; commit the new submodule SHA
  in litepdf. `.gitmodules` unchanged (`ignore = dirty` stays).
- `cmake/ImportMuPDF.cmake`: `_PRUNE_VER` → `LITEPDF_PRUNE_V3`. Confirm 1.27's
  `config.h` still uses `#ifndef`-guarded `FZ_ENABLE_*` **and** the bare-`#ifndef`
  `TOFU_*` pattern in `font-table.h`/`noto.c` (both verified in the spike — the
  prune depends on both halves); adjust the injected set if 1.27 renamed/moved any
  macro. The restore-pristine-via-`git checkout -- config.h` logic stays. Add any
  extra prune defines the spike found necessary to hold under 19 MB. **Add a
  configure-time assertion that each injected macro is actually present/effective
  after injection** (Opus F6 + Codex #5) — a silent macro-drift no-op would
  otherwise surface only as a late, ambiguous size-gate failure.
- **`src/` unchanged.** `page_hits` keeps calling `fz_search_page`; the
  `(void)flags; (void)abort_flag;` no-ops remain. Search output is identical to
  today.

**Determinism:** if the spike flagged a shift, regenerate the reference fixtures
under 1.27 **after** a human reviews the pixel diff of **all** fixture pages —
Latin (`large.pdf`) as well as CJK, not just a CJK no-tofu glance (Sonnet I2: an
AA shift on Latin text could otherwise slip through a CJK-only check). The PR
must carry before/after screenshot pairs for every regenerated reference and
document the shift. If no shift, fixtures are untouched.

**Tripwire stays armed:** `test_document_search.cpp`'s `[!shouldfail]`
case-sensitive test remains green in Stage 2 — search code is unchanged, so the
case-sensitive assertion still "fails as expected" and Catch2 keeps it green. It
only flips in Stage 3.

**Gate:** 173 unit tests pass; `litepdf.exe` < 19,000,000 B; determinism
`--check` green (against possibly-regenerated references); CJK renders real
glyphs; benchmark self-test 8/8; three-lens PR-merge review.

## 6. Stage 3 — Tier-1 activation

**`Document::page_hits` (`src/core/Document.cpp`, the `fz_search_page` block at
~489-498):**

**Implement to the collector shape + signatures the Stage 1 spike confirmed**
(3-lens High) — the bullets below state the *intended* mapping, but if the spike
recorded a different callback signature or hit-collection shape, that recording
wins over this text.

- The whole stext extraction + match runs **inside the existing `doc_mutex`
  lock** already taken at `Document.cpp:465` (NOT a new lock — Sonnet's C2 alarm
  that `page_hits` is unlocked was verified false; the lock is there, `outline()`
  at `:541` is the reference pattern). Extract an `fz_stext_page` from the page
  (`fz_new_stext_page_from_page` or via the display list — exact path from the
  spike), then call the 1.27 matcher.
- Build `fz_search_options` from `flags.match_case`: `true` → `FZ_SEARCH_EXACT`,
  `false` → `FZ_SEARCH_IGNORE_CASE`.
- **Mid-page cancellation:** via the callback form (intended:
  `fz_match_stext_page_cb` returning 1 when `abort_flag` is non-zero — confirm
  the real signature from the spike). Remove the two `(void)` no-ops.
- **Hit collection:** fill `PageHit` (one quad per hit) and preserve the
  `kMaxQuads = 256` cap + its truncation log (`Document.cpp:514-518`) on the new
  matcher's result shape, per the spike's prototype collector.
- **Refcount discipline:** drop the `fz_stext_page` in `fz_always`
  (`fz_drop_stext_page`), within the `doc_mutex` scope. This work follows the
  project's validated `mupdf-refcount-conventions` skill.
- `PageHit::snippet_utf8` still returns the needle itself (centered snippet is
  phase-6.2 / Tier 2, deferred).

**`src/core/SearchSession.cpp`:** the `abort_flag` pass-through to `page_hits`
(the call site, `~:211-212`) is unchanged, but its per-page abort now takes real
effect. Verify both the epoch-bump cross-page cancellation and the new mid-page
cancellation behave.

**Tests (`tests/unit/test_document_search.cpp` — currently 7 `TEST_CASE`s):**

- Flip `page_hits: case-sensitive does not match mismatched case`: remove the
  `[!shouldfail]` tag, convert to a normal passing test (case-sensitive search
  for `"lorem"` against fixture `"Lorem"` returns empty).
- **Add a mid-page cancellation test with a discriminating contract** (Sonnet I1
  — a trivial "abort → empty → pass" proves nothing): on a multi-hit page,
  pre-set `abort_flag = 1` and assert the result is empty / fewer than the full
  count; then run the **same** page with `abort_flag = 0` (or `nullptr`) and
  assert the full count. The pair proves the abort path gates on the flag, not on
  a bug that always returns empty.
- Existing case-insensitive tests must still pass under `FZ_SEARCH_IGNORE_CASE`
  (page 0: 12 `Lorem` hits; lowercase `lorem`: 12 hits; CJK needle: ≥ 1 hit).

**Gate:** flipped + new search tests green; full unit suite (173 → **174** after
the added test) green; **a leak/refcount check on `fz_stext_page` under repeated
search** (the existing stress harness or an ASAN-equivalent run — the
refcount-conventions skill is guidance, not a test; Opus F3); three-lens review.

## 7. Error handling

- `fz_new_stext_page_from_page` / `fz_match_stext_page` run inside the existing
  `fz_try`/`fz_catch`; on failure `page_hits` logs to stderr and returns an empty
  result (same contract as today).
- Regex is out of scope, so `FZ_SEARCH_REGEXP` is never passed — no regex-syntax
  error path this effort.
- Per-page hit cap (`kMaxQuads = 256`) and its truncation log are preserved.

## 8. Testing strategy

Unit (Catch2): the 7 existing search `TEST_CASE`s — one of which (the
`[!shouldfail]` case-sensitive test) is flipped in place in Stage 3 — plus the
new mid-page-cancel test, taking the file to 8 search tests and the full suite
from **173 → 174**. Then: determinism `--check` (large + CJK fixtures) →
benchmark self-test 8/8 → size gate (< 19 MB). Pixel-diff review of all
regenerated fixture pages (not just a CJK glance) per §5.

## 9. Risks

| # | Risk | Mitigation |
|---|------|------------|
| R1 | 1.27 breaches the 19 MB size gate | D3: prune harder to claw back; can't ⇒ defer (do not raise the gate). |
| R2 | Render determinism shifts under 1.27 | Stage 2 regenerates reference fixtures after CJK visual confirm; documented in PR. |
| R3 | Experimental stext API unstable / signature differs | Pin a specific 1.27.x tag; MuPDF types are already isolated behind Document's MuPDF-free header; spike catches signature divergence before any commitment. |
| R4 | stext extraction cost per `page_hits` | `fz_search_page` already extracts stext internally, so cost is **expected** ~unchanged — but the explicit `fz_new_stext_page_from_page` path could double-extract or use heavier options; treat as unconfirmed until the Stage 3 benchmark search-latency metric verifies it (Opus F2). |
| R5 | Refcount / cross-thread mishandling of `fz_stext_page` | Per-call lifetime, dropped in `fz_always` under `doc_mutex`; follow `mupdf-refcount-conventions`. |
| R6 | Prune adaptation stacks two layers / breaks on 1.27 config.h | `ImportMuPDF.cmake` already restores pristine `config.h` before re-injecting and FATAL_ERRORs on a failed restore; spike verifies the 1.27 config.h structure. |

## 10. Out of scope

- Tier 2 search: regex, whole-word, FindBar UI toggles, centered snippet.
- MuPDF 8 MB exe target (post-v1.0, needs a Windows system-font loader).
- DPI-/CPU-count-adaptive `SearchDispatcher` sizing (separate post-v1.0 item).
- Re-opening the "execute post-v1.0" timing: this spec produces the design now;
  whether Stages 2-3 land before or after v1.0 ships is gated on the spike's
  go/no-go and a separate user call.
